The Blocker

p4 zip aborts on partial move actions

When a changelist contains a move/add or move/delete whose counterpart sits outside the command's view, p4 zip stops with:

"the parameters of this fetch, push, or zip
 command include only part of the full action"

The fix that works: ensure the remote spec covers both halves of every move. Doing that by hand at 705k paths is not feasible. We automate it.

Changelist 142,857 ───────────────────────────────── move/add //depot/new/foo.c move/delete //depot/old/foo.c ← outside view ✗ ┌────────────────────┐ │ p4 zip │ │ view = //depot/new│ └─────────┬──────────┘ ▼ ╳ HALT — partial action

Validation Without Perforce Sign-Off

Three independent gates prove the migration is faithful

Gate 1 · Pre-zip

Spec coverage proof

  • Total view lines ≥ 2 × distinct moves discovered
  • Random 100-pair manual inspection (both halves present)
  • p4 files //depot/... count comparable to view size

Gate 2 · Post-unzip

Server-level verify

  • p4 verify -q //depot/... on destination — zero BAD/MISSING
  • Changelist count parity (source vs. destination)
  • Revision count parity per top-level branch

Gate 3 · End-to-end

Sample diff

  • Pick 50 files at random across history
  • p4 print -q + SHA256 on both sides
  • All 50 hashes must match — no exceptions

If any gate fails, the migration does not promote. The destination depot is created fresh per attempt; the source is read-only from T+08:00. Failure is reversible up to that point.

Risks & Mitigations · 1 of 2

Technical risks of running our own tool

Move chains (A→B→C across changelists)
Medium
Tool may capture only the most recent movedFile, missing intermediate hops. Result: p4 zip still fails on a chain.
Mitigation: after first scan, run a transitive-closure SQL pass over moves.sqlite — for every (src,tgt) row, recursively pull any other row whose tgt = our src. Add the transitive paths to the spec. Re-zip if needed.
Branch-of-move & integration-then-move
Medium
Some branched-and-then-moved files surface as branch+add+delete, not move/*. Tool would miss these.
Mitigation: on first p4 zip failure, the tool already returns the failing changelist. Run p4 describe -s on it, add any referenced paths to the spec, re-zip. Empirically converges within 2–3 iterations.
Lazy copies / integration history
Low
Lazy-copy attribution lives in integration records, not filelog. Could in principle widen the required view further.
Mitigation: if the iterative loop above doesn't converge after 5 retries, fall back to a single broad spec (//depot/... //remote/...) and let p4 zip use the entire depot. Slower zip, but guarantees coverage.
Obliterated halves of moves
Low
If a move's other half was obliterated, movedFile is NULL → tool emits a one-sided row → no harm to spec, but no new coverage either.
Mitigation: these cases will not block p4 zip because Perforce already treats obliterated revisions as gone. Detected during Gate 2 verify.
Server load from parallel filelog
Medium
8 parallel readers on a production server during business hours could degrade response.
Mitigation: default to --workers 4 not 8 · run scan in off-peak window · monitor server with p4 monitor show · throttle to --workers 2 if any p99 latency spike observed.
SQLite cache corruption mid-run
Low
Power loss or process kill could leave the cache in a partial state.
Mitigation: SQLite is ACID by default · we commit per range · UNIQUE constraint dedupes restarts · worst case: delete cache and restart scan (idempotent).

Risks & Mitigations · 2 of 2

Operational & commercial risks

No vendor sign-off on the workaround
Medium
If the migrated depot later shows an issue, Perforce support may decline to debug because the depot was produced by a non-Perforce tool path.
Mitigation: our tool only generates a remote spec and runs the official p4 zip binary — output format is identical to a hand-built spec run. Document the exact spec used, archive it, share with support if a case is opened.
Future p4 zip behaviour change
Low
A future Perforce release could deprecate or alter remote-spec format, breaking re-runs.
Mitigation: we only need this tool for 2 more migrations. After that the depots live on the destination; the tool is retired or replaced by Perforce's official fix when available.
Data loss during zip → unzip cycle
High
Worst-case scenario: corrupted zip is unzipped onto destination, source decommissioned before discovery.
Mitigation: source server is set to read-only, not decommissioned, until ALL THREE depots have passed Gate 3 sample-diff. Decommission only after end-of-project sign-off.
Tool maintenance burden
Low
Custom tooling means we own bugs and maintenance.
Mitigation: 400 LoC, MIT-licensed, 16 tests, single Python dependency (p4python). Maintainable in-house. Drop the moment Perforce ships the fix.
Customer (Amdocs) trust
Medium
Amdocs may question why we chose a custom tool over waiting for Perforce.
Mitigation: deliver this exact deck to Amdocs · make the source readable · show passing tests · offer to demo live · point to the documented validation gates · their migration deadline beats the vendor's roadmap.

Run-book

The 10 commands we run today

  1. Snapshot: back up /p4/2/root (storage-level snapshot)
  2. Capture HEAD: p4 -p illin2343:1666 counter change
  3. Install tool: pip install -e "p4-move-zip-fixer[p4]"
  4. Authenticate: P4PORT=illin2343:1666 p4 login
  5. Scan: p4-move-zip-fixer scan --depot //depot/... --head $HEAD --workers 4 --db moves.sqlite
  6. Build spec: p4-move-zip-fixer build-spec --db moves.sqlite --remote migration-remote
  7. Gate 1 inspect: p4 remote -o migration-remote & sample-check
  8. Zip: p4-move-zip-fixer zip --remote migration-remote --output default-depot.zip  (loop on failed changelists if any)
  9. Transfer & unzip: scp default-depot.zip p4jirastg22:/tmp/ && p4 unzip -i /tmp/default-depot.zip
  10. Gate 2 + 3: p4 verify -q //depot/... · changelist parity · 50-file SHA256 sample diff

All commands logged  Spec archived  SQLite kept — full audit trail for later support, cases, or repeat migrations.
